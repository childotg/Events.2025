OAM Approach

apiVersion: core.oam.dev/v1beta1
kind: Application
metadata:
  name: my-api
spec:
  components:
  - name: api
    type: webservice
    properties:
      image: myimage:v1
      port: 8080
    traits:
    - type: scaler
      properties:
        replicas: 3

✓ Platform-agnostic, vendor-neutral, separation of concerns

- "OAM looks clean: describe WHAT you want, not HOW"
- "Platform-agnostic, vendor-neutral—sounds perfect"
- "But there's a problem..."

The Meta-Language Trap

❌ Superset complexity: must support all clouds' features

❌ Learning curve: new spec to learn, evolving standard

❌ Tooling immaturity: ACA not natively supported

❌ Double abstraction: App → OAM → K8s → Container

You're learning a language to describe another language

- Read bullets quickly, emphasize:
- "Superset complexity: OAM must model ALL clouds' features"
- "Learning curve: new spec, evolving (v1alpha2 → v1beta1)"
- "ACA not natively supported—you'd write custom traits anyway"
- Key phrase: "You're learning a language to describe another language"
- PAUSE 2 seconds

When to Use ACA, When to Migrate

Scenario Strategy Why
Dev/staging environments ✅ Use ACA Scale-to-zero saves costs, rapid iteration
Simple stateless APIs ✅ Use ACA Serverless fits perfectly, less overhead
Compliance / data sovereignty 🚀 Migrate to K8s Deploy on-prem or edge clusters as needed
Multi-cloud strategy 🚀 Migrate to K8s Run on Azure (AKS), AWS (EKS), GCP (GKE)
Complex networking / control 🚀 Migrate to K8s Full control plane, custom policies, advanced features
True multi-cloud from day one 🤔 Consider OAM If deploying to AWS, GCP, Azure simultaneously—but expect complexity
Azure-first with portability ✅ K8s + AI Start with ACA/AKS, maintain K8s manifests, AI when you need to escape

Recommended: K8s + AI for Azure-first with escape hatch

Your Next Steps (Within 2 Weeks)

2. Use AI: "Generate K8s manifests from this ContainerApp"
3. Test on any K8s cluster (minikube, AKS, etc.)

5. Validate bidirectional transformation works both ways

If You're Starting Fresh:

1. Create K8s manifests first (Deployment, Service, Ingress)
2. Use AI: "Generate ACA YAML from these K8s manifests"
3. Deploy to ACA if serverless makes sense
4. Test: deploy same K8s source to any cluster
5. Keep K8s as source of truth going forward
